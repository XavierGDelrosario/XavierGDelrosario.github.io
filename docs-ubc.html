UBC Course Query

Documentation for a full-stack query engine over UBC course and room data — the dataset model, the JSON query language, the API contract, and the agile process behind it.

Overview

  • A web application that ingests UBC course datasets, stores them, and answers structured queries over them — returning results as JSON and as interactive bar charts.
  • Datasets are uploaded as zip archives, parsed into an internal model, and persisted to disk so they survive restarts and can be re-loaded into memory to answer queries.
  • Exposed both as a programmatic InsightFacade API and over RESTful HTTPS endpoints.

Datasets

  • Two kinds are supported: sections (course sections — department, id, average, instructor, etc.) and rooms (building and room metadata).
  • Each dataset has a unique id. An id is rejected if it contains an underscore or is only whitespace (it must match /^[^_]+$/), or if it duplicates an existing id.
  • Content is supplied as a base64-encoded zip; rows are counted and reported back via listDatasets().

Query language

Queries are plain JSON. A WHERE block selects rows; an OPTIONS block chooses columns and ordering. Below: departments and averages for sections with an average above 97, sorted ascending.

// SELECT dept, avg WHERE avg > 97
{
  "WHERE":   { "GT": { "sections_avg": 97 } },
  "OPTIONS": {
    "COLUMNS": [ "sections_dept", "sections_avg" ],
    "ORDER":   "sections_avg"
  }
}
  • Filters — numeric comparisons GT / LT / EQ, string matching IS (with * wildcards), and logic AND / OR / NOT, nestable to any depth.
  • Aggregation — group rows and apply MAX / MIN / AVG / SUM / COUNT to a field.
  • Sorting — stable ordering on one or more columns, ascending or descending.
  • A malformed query, or one referencing a dataset that wasn't added, is rejected with an InsightError; a query returning too many rows is rejected with a ResultTooLargeError.

API surface

  • addDataset(id, content, kind) → resolves to the ids of all currently added datasets.
  • removeDataset(id) → resolves to the removed id, or rejects with NotFoundError if it was never added.
  • performQuery(query) → resolves to an array of result rows.
  • listDatasets() → resolves to each dataset's id, kind, and row count.

Tech stack

  • Backend — Node.js with an Express HTTPS server; datasets persisted to disk as JSON.
  • Language & tooling — TypeScript, with ESLint and Prettier enforced on every build.
  • Testing — Mocha + Chai, including chai-http tests that exercise full request–response cycles against the live endpoints.

How it was planned

  • Tracked work as GitHub Issues using a structured User Story template — each story captured a Role / Goal / Benefit, a Definition of Done, a breakdown of engineering tasks, and a time estimate.
  • Defined the query grammar and the RESTful API contract before implementation, so the frontend and backend could be built against an agreed interface.
  • Test-first: query behaviours were validated with a Mocha/Chai suite mirroring the spec.
